# 如何约束 Claude 按需求文档精确执行

> 基于 car-content-producer 项目实战总结
> 日期：2026-03-25

---

## 一、为什么 Claude 会偏离需求文档？

### 原因 1：文档是「描述性」的，Claude 把它当参考而非约束

Claude 在实现时会做工程判断：
- `match_method` 文档写了四个枚举值，Claude 觉得「等价的」就自己换了三个
- `generate_audio()` 文档说返回字符串，Claude 觉得返回 bool「更自然」
- 搜索 URL 文档写了 `/auto/library/series?kw=`，Claude 觉得 `/search?keyword=` 效果一样

**根本原因**：文档没有显式写「这是强制要求，不能改」，Claude 就当成「例子」。

### 原因 2：增量实现导致跨模块接口遗忘

Claude 按顺序写 storage → resolver → scraper → writer → video → main。
写 main.py 时已不记得 video.py 返回 bool 还是 string，直接假设是字符串（或反过来）。
这是跨模块接口对齐问题，不是单文件问题。

### 原因 3：文档的隐含信息 Claude 会忽略

`raw_materials.json` 写在「输出目录结构」图里，但没有独立章节说「这是必须输出」。
Claude 读到的优先级：有独立章节 > 列表里出现 > 目录结构图。

## 二、实战发现的偏差清单（本项目）

| 偏差 | 原因 | 修复 |
|------|------|------|
| `generate_audio()` 返回 `bool` 而非字符串 | 文档没强调类型 | 加强约束语言 |
| `topic.json` 而非 `final_topic.json` | 文件名未强制 | 改名 + 加入必须输出列表 |
| `raw_materials.json` 缺失 | 隐含在目录图里 | 加入「必须输出」章节 |
| 热点搜索用 Playwright 而非 Tavily | 「用 Tavily」写法太弱 | 改为「禁止用 Playwright 作为主要来源」 |
| `match_method` 枚举值不符 | 文档未写「只允许」 | 加「只允许以下四个值」 |
| `material_count` 从未赋值 | 文档未列 set_field 调用时机 | 明确每步 set_field 调用 |
| errors 格式缺 `level` 字段 | 文档示例未强调 | 加「必须包含」 |

## 三、下次如何约束 Claude 精确执行

### 方法 1：在文档里明确区分「强制」和「参考」

```markdown
## 强制输出文件（缺少任何一个视为失败）
- [ ] data/final_topic.json
- [ ] data/raw_materials.json
- [ ] data/scraped_data.json
- [ ] data/article.json
- [ ] data/subtitles.srt
- [ ] 知乎_*.html
- [ ] youtube_script_*.md
- [ ] metadata.json

## 强制接口契约（不允许「等价替换」）
- generate_audio() 返回值：必须是字符串 "ok" | "fallback_edge_tts" | "failed"，禁止返回 bool
- match_method：只允许 "exact_name" | "search_page" | "fuzzy_brand" | "failed"
- errors 条目：必须包含 step, level, message, retry_count, degraded 五个字段
```

### 方法 2：用「禁止」语言而非「应该」语言

| 弱约束（容易被忽略） | 强约束（Claude 会遵守） |
|---|---|
| `match_method` 建议用以下枚举 | `match_method` **只能是**以下四个值，其他值视为 bug |
| 热点搜索用 Tavily API | **禁止**用 Playwright 做热点搜索主要来源，必须用 Tavily API |
| 输出 raw_materials.json | **每次运行必须**输出 raw_materials.json，缺少则程序报错退出 |
| TTS 返回成功/失败 | TTS 函数**必须返回字符串**，不得返回 bool，违反则视为接口错误 |

### 方法 3：让 Claude 先写接口契约再写实现（最有效）

**第一步**：只根据技术方案文档，写出所有函数签名（函数名、参数类型、返回值类型），**不写实现**。

```
请根据技术方案文档，只写出所有模块的函数签名，格式如下：
def function_name(param: Type) -> ReturnType
    """一句话说明"""
不要写任何实现代码。
```

**第二步**：我确认签名后，再写实现。

这样能在写代码之前就暴露接口理解偏差，避免写完后才发现类型不对。

### 方法 4：写完代码后要求 Claude 做 checklist 验证

```
你刚写完代码，现在逐条对照需求文档做验证：
1. 所有「强制输出文件」是否都有对应的 save_json 调用？列出每个文件的调用位置。
2. 所有函数返回值类型是否与函数签名契约完全一致？
3. 所有 URL、枚举值是否与文档完全相同（禁止「等价替换」）？
4. metadata.json 的 errors/warnings 条目格式是否包含所有必填字段？
如有不符，立即修复并说明修改了什么。
```

### 方法 5：把「技术方案文档」第一节改为「约束声明」

在技术方案文档最开头加：

```markdown
## 零、实现约束声明（必读，优先级高于所有其他章节）

以下约束在实现时不允许以任何理由偏离：
1. 所有函数签名见「第十六节」，不允许修改参数名、类型、返回值类型
2. 所有输出文件名见「第四节必须输出列表」，不允许修改文件名
3. 所有枚举值见「第五节数据模型」，不允许增减或重命名
4. 禁止 AI 编造参数数据，所有车辆参数必须来自 scraper.py 抓取结果
5. 实现完成后必须自行执行第四节的「checklist 验证」
```

## 四、Prompt 模板：让 Claude 按需求精确实现

### 开始实现前

```
我有一份技术方案文档（附后）。在你写任何代码之前：
1. 先列出文档中所有「强制要求」（用「必须」「禁止」「只允许」标记的）
2. 列出所有函数签名契约
3. 列出所有必须输出的文件
4. 我确认后你再开始写实现
```

### 实现完成后

```
代码写完后，逐条验证：
- 每个强制输出文件是否有对应代码？
- 每个函数返回类型是否与契约一致？
- 每个枚举值是否与文档完全相同？
发现不符立即修复，不要等我指出。
```

### 修改现有代码前

```
在修改这个函数前，先读一遍技术方案文档中关于它的签名契约。
修改后返回类型必须与契约一致，不允许因为「更自然」而改变类型。
```

## 五、总结

Claude 偏离需求的核心原因是：**文档写的是「意图」，Claude 在实现时会做「工程判断」来替代意图**。

解决方法不是让 Claude「更认真读文档」，而是：
1. **把「意图」转化为「约束」**：用禁止、只允许、必须等强制语言
2. **把「例子」转化为「契约」**：函数签名章节明确类型，不能等价替换
3. **把「验证」变成流程的一部分**：写完代码必须做 checklist，不是可选的
